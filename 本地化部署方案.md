# CRM 系统本地化部署方案

## 一、核心结论

**你的 CRM 系统本身就是为本地部署设计的**（SQLite 数据库 + Node.js 后端），不需要大改。关键是要建立**自动化运维体系**。

---

## 二、本地部署 vs 云服务器对比

| 维度 | 云服务器 | 本地部署 | 解决方案 |
|------|----------|----------|----------|
| **成本** | 高（持续付费） | 低（一次性硬件） | ✅ 本地更优 |
| **数据安全** | 依赖云厂商 | 完全自控 | ✅ 本地更优 |
| **稳定性** | 高（专业运维） | 需自己维护 | ⚠️ 需自动化 |
| **远程访问** | 天然支持 | 需配置 | ⚠️ 需内网穿透 |
| **维护难度** | 低 | 中 | ⚠️ 需自动化脚本 |

## 三、稳定性保障方案（不会容易崩）

### 3.1 硬件层面

```
推荐配置：
- CPU: 4 核以上（Intel i5 或同级）
- 内存：8GB 以上
- 存储：SSD 256GB + 机械硬盘 1TB（数据库放 SSD，备份放机械盘）
- 网络：固定公网 IP 或动态 DNS
- UPS：不间断电源（防止断电损坏数据库）
```

### 3.2 系统层面

#### ✅ Docker 容器化部署（强烈推荐）

**为什么用 Docker：**
- 环境隔离，避免系统更新导致依赖冲突
- 一键启动/重启，崩溃后自动恢复
- 便于备份和迁移

**docker-compose.yml 示例：**

```yaml
version: '3.8'

services:
  crm-backend:
    build: ./backend
    container_name: crm-backend
    restart: always  # 崩溃自动重启
    ports:
      - "3000:3000"
    volumes:
      - ./backend/data:/app/data  # 数据库持久化
      - ./backend/logs:/app/logs  # 日志持久化
      - ./backups:/backups  # 备份目录
    environment:
      - NODE_ENV=production
      - DB_PATH=/app/data/crm.db
    healthcheck:  # 健康检查
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s

  crm-frontend:
    build: ./frontend
    container_name: crm-frontend
    restart: always
    ports:
      - "80:80"
    depends_on:
      - crm-backend
```

### 3.3 自动备份方案

#### ✅ 三重备份策略

```bash
#!/bin/bash
# backup-crm.sh - 每日自动备份脚本

BACKUP_DIR="/mnt/backup/crm"
DATE=$(date +%Y%m%d_%H%M%S)
DB_PATH="/root/.openclaw/workspace/crm-system/backend/data/crm.db"

# 1. 本地备份（保留 7 天）
cp $DB_PATH $BACKUP_DIR/daily/crm_$DATE.db

# 2. 异地备份（保留 30 天）- 可同步到 NAS 或另一台机器
rsync -av $DB_PATH user@backup-server:/backup/crm/

# 3. 云备份（保留 90 天）- 可选阿里云 OSS/腾讯云 COS
ossutil cp $BACKUP_DIR/daily/crm_$DATE.db oss://your-bucket/crm-backup/

# 清理 7 天前的本地备份
find $BACKUP_DIR/daily -name "crm_*.db" -mtime +7 -delete

echo "[$(date)] 备份完成：crm_$DATE.db" >> /var/log/crm-backup.log
```

**添加到 crontab：**

```bash
# 每天凌晨 2 点自动备份
0 2 * * * /root/.openclaw/workspace/crm-system/scripts/backup-crm.sh
```

### 3.4 自动监控告警

#### ✅ 系统健康监控脚本

```bash
#!/bin/bash
# monitor-crm.sh - 系统健康检查

# 检查后端服务
if ! curl -sf http://localhost:3000/health > /dev/null; then
    echo "[$(date)] 警告：CRM 后端服务异常" | mail -s "CRM 告警" your@email.com
    systemctl restart crm-backend
fi

# 检查磁盘空间
DISK_USAGE=$(df / | tail -1 | awk '{print $5}' | sed 's/%//')
if [ $DISK_USAGE -gt 85 ]; then
    echo "[$(date)] 警告：磁盘使用率 ${DISK_USAGE}%" | mail -s "CRM 告警" your@email.com
fi

# 检查数据库完整性
sqlite3 /root/.openclaw/workspace/crm-system/backend/data/crm.db "PRAGMA integrity_check;"
if [ $? -ne 0 ]; then
    echo "[$(date)] 严重：数据库损坏！" | mail -s "CRM 告警" your@email.com
fi
```

**每 5 分钟检查一次：**

```bash
*/5 * * * * /root/.openclaw/workspace/crm-system/scripts/monitor-crm.sh
```

## 四、远程访问方案（在家/外出也能访问）

### 4.1 方案对比

| 方案 | 难度 | 成本 | 安全性 | 推荐度 |
|------|------|------|--------|--------|
| 固定公网 IP+ 端口映射 | 中 | 低 | 中 | ⭐⭐⭐ |
| 内网穿透（frp） | 中 | 低 | 高 | ⭐⭐⭐⭐ |
| Tailscale/ZeroTier | 低 | 免费 | 高 | ⭐⭐⭐⭐⭐ |
| 云服务器反向代理 | 低 | 中 | 高 | ⭐⭐⭐⭐ |

### 4.2 推荐方案：Tailscale（最简单）

**步骤：**

```bash
# 1. 在本地服务器安装 Tailscale
curl -fsSL https://tailscale.com/install.sh | sh

# 2. 登录激活
sudo tailscale up

# 3. 在手机/笔记本安装 Tailscale 客户端
# 4. 登录同一账号，即可通过 Tailscale IP 访问 CRM

# 访问地址：https://100.x.x.x:3000
```

**优点：**
- 无需公网 IP
- 端到端加密
- 免费个人使用
- 配置简单（3 步完成）

### 4.3 备选方案：frp 内网穿透

**需要一台云服务器作为中转：**

```ini
# frps.ini（云服务器配置）
[common]
bind_port = 7000
token = your_secret_token

# frpc.ini（本地服务器配置）
[common]
server_addr = your-cloud-server-ip
server_port = 7000
token = your_secret_token

[crm]
type = tcp
local_ip = 127.0.0.1
local_port = 3000
remote_port = 6000
```

**访问地址：** `http://your-cloud-server-ip:6000`

## 六、崩溃恢复预案

### 6.1 自动重启（Docker 已配置）

```yaml
restart: always  # 容器崩溃自动重启
```

### 6.2 手动恢复流程

```bash
# 1. 查看日志定位问题
docker logs crm-backend --tail 100

# 2. 尝试重启服务
docker-compose restart

# 3. 如果数据库损坏，从备份恢复
cp /mnt/backup/crm/daily/crm_20260328.db /opt/crm/data/crm.db
docker-compose restart

# 4. 验证恢复
curl http://localhost:3000/health
```

### 6.3 灾难恢复（硬件损坏）

```bash
# 1. 准备新服务器
# 2. 安装 Docker
# 3. 从备份恢复数据
scp user@backup-server:/backup/crm/crm_latest.db ./data/
# 4. 启动服务
docker-compose up -d
# 5. 验证服务
curl http://localhost:3000/health
```

**目标：30 分钟内恢复服务**

## 九、常见问题解答

### Q1: 断电怎么办？
**A:** UPS 可提供 15-30 分钟缓冲时间，配合自动关机脚本安全关闭。来电后 Docker 会自动重启服务。

### Q2: 硬盘损坏怎么办？
**A:** 三重备份策略确保数据安全。本地备份（7 天）+ 异地备份（30 天）+ 云备份（90 天），最多丢失 1 天数据。

### Q3: 网络断了怎么办？
**A:** 本地使用不受影响。远程访问可通过备用方案（云服务器反向代理）临时切换。

### Q4: 系统更新怎么办？
**A:** Docker 镜像更新流程：
```bash
docker-compose pull  # 拉取新镜像
docker-compose up -d  # 重启容器（自动迁移数据）
```

### Q5: 我不懂技术怎么办？
**A:** 所有脚本都已自动化，日常无需操作。告警会发送到你的钉钉/邮箱，需要处理时我会远程协助。
